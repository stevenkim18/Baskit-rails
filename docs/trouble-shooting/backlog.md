# 인증 구현 후속 백로그

로그인 기능 1차 구현 이후, 후속으로 해결하거나 정책을 확정해야 할 항목을 백로그 형태로 정리한다.

## 항목 1. 로그아웃 범위 정책 확정

- 우선순위: `P1`
- 현재 상태: `DELETE /api/v1/auth/session` 요청에서 `refresh_token`이 없으면 현재 사용자의 모든 활성 refresh token을 revoke한다.
- 영향:
  - 현재 기기 로그아웃이 아니라 전체 기기 로그아웃처럼 동작할 수 있다.
  - iOS가 기대하는 UX와 다르면 연동 시 혼선이 생긴다.
- 해결해야 할 질문:
  - 기본 로그아웃은 현재 기기만 처리할지
  - 전체 기기 로그아웃은 별도 API로 분리할지
  - iOS 앱이 어떤 동작을 기대하는지
- 권장 다음 액션:
  - API 정책 먼저 확정
  - 필요하면 `DELETE /api/v1/auth/session` 계약 변경
  - 전체 로그아웃 전용 엔드포인트 추가 여부 검토

## 항목 2. 회원가입 동시성 테스트 보강

- 우선순위: `P2`
- 현재 상태: `RecordNotUnique` 예외 처리는 구현했지만, 테스트는 `User.create!` stub 기반이다.
- 영향:
  - 컨트롤러 예외 처리는 검증되지만 실제 병렬 요청 환경을 완전히 재현한 것은 아니다.
  - 운영에서 매우 짧은 간격의 중복 회원가입 요청이 들어오면 예상치 못한 변형 케이스가 남을 수 있다.
- 해결해야 할 질문:
  - 병렬 요청을 재현하는 통합 테스트가 필요한지
  - 현재 수준에서 운영 모니터링으로 충분한지
- 권장 다음 액션:
  - 가능하면 동시 요청 통합 테스트 추가
  - 최소한 중복 회원가입 실패 로그를 남기도록 검토

## 항목 3. Swagger 환경별 설정 정리

- 우선순위: `P3`
- 현재 상태: 로컬 Swagger UI는 `/api-docs`에서 확인 가능하고, OpenAPI `servers` 값은 `http://localhost:3000` 기준이다.
- 영향:
  - 스테이징/운영 환경에서는 잘못된 서버 URL이 문서에 남을 수 있다.
  - 환경이 바뀔 때마다 Swagger 문서를 다시 생성해야 할 수 있다.
- 해결해야 할 질문:
  - 환경별 `servers` 값을 어떻게 분리할지
  - `rswag` deprecation warning을 언제 정리할지
- 권장 다음 액션:
  - 개발/스테이징/운영별 Swagger server 설정 분리
  - `rswag` 설정 업데이트로 deprecation warning 제거
